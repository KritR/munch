# Install

## Purpose and scope

This document defines how users install `munch`, connect it to their shell, and verify that the setup works.

It covers:

* binary placement
* CLI-assisted shell setup
* embedded shell init scripts
* config bootstrap
* post-install verification

It does not define release artifact generation or CI/CD. Those belong in `docs/packaging.md`.

## Goals

The install model should:

* keep the shipped binary name as `munch`
* support manual installation from release artifacts
* use Starship-style shell setup through `munch init zsh` and `munch init fish`
* keep shell integration self-contained inside the binary
* avoid mutating shell rc files automatically
* use `urfave/cli` for command routing, option handling, help text, and completion generation

## Non-goals

The initial install flow does not aim to provide:

* automatic shell rc file edits
* a required curl-pipe-sh installer
* package-manager-specific instructions in iteration 1
* installing shell integration files into XDG data directories

## Install model

Iteration 1 install is split into two parts:

1. place the `munch` binary on `PATH`
2. add a shell init line that evaluates shell code generated by the CLI

This means the CLI should support:

* `munch init zsh`
* `munch init fish`
* `munch init zsh --key <sequence>`
* `munch init fish --key <sequence>`
* `munch init <shell> --no-bind`

Those commands should print executable shell integration code to stdout. The user's shell config should evaluate or source that generated code at startup.

This follows the same operating model as Starship's shell setup, where the installed binary owns the shell-specific init script and users add a small shell-native invocation to their rc file.

## Binary placement

The installed binary should be named:

* `munch`

The binary is expected to live in a user-accessible directory already on `PATH`, or in a directory the user chooses to add to `PATH`.

The exact installation command depends on the chosen distribution method, but the user-facing contract should remain stable:

* the executable they run is `munch`

## CLI framework

The CLI should use `urfave/cli/v3`.

The initial command surface should include:

* `munch`
* `munch init zsh`
* `munch init fish`
* `munch init <shell> --key <sequence>`
* `munch init <shell> --no-bind`
* `munch completion <shell>` if completion generation is added in the same phase
* `munch --version`

The CLI should own shared flags such as:

* `--config`
* `--help`
* `--version`

Widget/runtime-specific options should remain available to the shell bridge, but they should not dominate the user-facing help output.

## Embedded shell init scripts

Shell integration code should be embedded into the Go binary rather than installed as separate files.

The implementation should keep one source file per supported shell in the repository, then embed those files with Go's `embed` package at build time.

Recommended source layout:

* `internal/shellinit/scripts/munch.zsh`
* `internal/shellinit/scripts/munch.fish`

Recommended Go package:

* `internal/shellinit`

The package should expose a small API, for example:

```go
func Script(shell string) (string, error)
```

`munch init <shell>` should print the embedded script for that shell directly to stdout.

This keeps installation self-contained:

* no shell files are written during install
* no runtime lookup path is required for shell assets
* upgrading the binary upgrades the shell integration code used by new shells
* uninstalling the binary and removing the rc snippet fully removes the integration

## Zsh setup

`munch init zsh` should print executable Zsh code.

Users should add this line to `~/.zshrc`:

```zsh
eval "$(munch init zsh)"
```

The generated Zsh code should:

* define the ZLE widget
* bind the widget to the chosen key sequence unless `--no-bind` is set
* call back into `munch` for bridge/runtime execution
* restore or replace the shell buffer based on the response action

The generated code should assume `munch` is available on `PATH`.

The default Zsh binding should be `^G`. Users can override it in their rc file:

```zsh
eval "$(munch init zsh --key '^X^M')"
```

Advanced users can opt out of automatic binding:

```zsh
eval "$(munch init zsh --no-bind)"
```

## Fish setup

`munch init fish` should print executable Fish code.

Users should add this line to `~/.config/fish/config.fish`:

```fish
munch init fish | source
```

The generated Fish code should:

* define the Fish binding function
* bind the widget in the relevant Fish key maps unless `--no-bind` is set
* call back into `munch` for bridge/runtime execution
* restore or replace the shell buffer based on the response action

The generated code should assume `munch` is available on `PATH`.

The default Fish binding should be `\cg`. Users can override it in their Fish config:

```fish
munch init fish --key \cx\cm | source
```

Advanced users can opt out of automatic binding:

```fish
munch init fish --no-bind | source
```

## Config bootstrap

The config file location should remain:

* `$XDG_CONFIG_HOME/munch/config.toml` when `XDG_CONFIG_HOME` is set
* fallback `~/.config/munch/config.toml`

Install/setup guidance should include a minimal config example for the first provider.

## Provider environment variables

Provider credentials should continue to come from environment variables rather than storing secrets in config.

For iteration 1, install instructions should include:

* the config key naming the provider env var
* the shell command needed to export that env var

## Post-install verification

The install flow should provide a simple verification sequence.

Recommended checks:

* verify the binary is runnable
* run `munch --version`
* run `munch init zsh` or `munch init fish`
* reload the shell
* trigger the widget once
* confirm the live UI opens and can return a command

## Upgrade and uninstall

Iteration 1 should document simple manual upgrade and uninstall behavior.

Upgrade:

* replace the binary with a newer release
* keep config and shell init snippets unchanged
* start a new shell so the rc file evaluates the new embedded init script

Uninstall:

* remove the `munch` binary
* remove the shell init snippet from shell config
* optionally remove the config directory
